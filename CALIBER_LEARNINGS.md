# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** The 4 docs that drift together when the renderer changes are `CLAUDE.md`, `AGENTS.md`, `README.md`, `DEVELOPER.md` — update all four. After any stack/architecture edit, grep them for stale terms (`react|zustand|axios|\.tsx|App\.tsx|VITE_API_URL`); e.g. `VITE_API_URL` was renamed `VITE_PHLIX_SERVER_URL`, and Hub config is runtime electron-store (`hub:set-config`), not an env var.
- **[convention]** `.{docs,migration,review,test}-worklog.md` are scratch artifacts — `rm -f` them before staging a commit; they are not meant to be committed.
- **[gotcha]** "Codacy Static Code Analysis" PR check can report `fail` but does NOT block merge — `master` is UNPROTECTED (`gh api repos/detain/phlix-windows-client/branches/master/protection` → 404). Codacy is advisory; the real gates are Lint / Type Check / Test on ubuntu-latest + windows-latest. `MERGEABLE`/`UNSTABLE` with only Codacy failing is safe to `gh pr merge --squash --delete-branch`.
- **[gotcha]** A backgrounded CI-poll loop exits with the last command's status, so when any check shows `fail` the loop's trailing `gh pr checks` exits non-zero and the `<task-notification>` marks the background command `failed` even when CI settled fine. Always `Read` the task output file before reacting — the "failed" status is just the trailing exit code.
- **[gotcha]** After `npm audit fix` reports 0 vulnerabilities locally, `gh api repos/.../dependabot/alerts` still shows alerts OPEN — Dependabot only closes them after the fix merges to the default branch and re-scans. Verify locally with `npm audit` + `npm ls <pkg>`, not the alert count.
- **[fix]** For Dependabot vite/vitest alerts here: old `vitest@1.x` + `@vitest/coverage-v8@1.x` transitively pull vulnerable `vite@5.4`. Bump both with `npm pkg set devDependencies.vitest="^3.2.6" devDependencies.@vitest/coverage-v8="^3.2.6"`, then `npm install` + `npm audit fix` (NOT `--force`) clears the chain to 0 vulns; suite, coverage, build, lint, vue-tsc stay green under vitest 3.
- **[pattern]** To clear deprecated transitive deps (`inflight` memory-leak warning, old `glob`/`rimraf`) without touching electron-builder, add an `overrides` block to `package.json` (`{"glob":"^13.0.6","rimraf":"^6.1.3"}`) and `npm install`. Verify with `npm ls inflight` → `(empty)`, then re-run `npx vitest run`. `npm warn deprecated glob@11.x` can still print during install even after the override resolves correctly — check the installed tree, not the warning.
- **[gotcha]** vitest coverage `exclude` in `vitest.config.mts` must cover ALL root config files via the glob `**/*.config.{ts,mts,js,mjs,cjs}`, not individually-named files — otherwise `vitest.config.mts`/`eslint.config.mjs` count as 0%-covered source and tank the "All files" total (48% instead of 100%).
- **[gotcha]** Windows CI `EBADENGINE` warnings (electron@42 / @electron/get / @electron/rebuild / @babel 8 want node `>=22.12`) are WARNINGS, not failures — `npm install` still succeeds. The fix is to bump the runner Node, not chase the install exit code.
- **[convention]** CI Node version is pinned in TWO files: `.github/workflows/test.yml` (matrix `node-version: [NN.x]` + lint/type-check jobs + the coverage-upload `if:` conditional that hard-codes the version string) and `.github/workflows/build.yml` (`NODE_VERSION: 'NN.x'` env). Bump BOTH. Node 24 is org-preferred (satisfies electron >=22.12 and @babel 8 / vitest 3).
- **[gotcha]** When bumping a `github:detain/phlix-ui#vX.Y.Z` (git-tag) dependency, `npm pkg set` + plain `npm install` can silently KEEP the old version (lockfile/cache resolves the old tag) — verify with `node -p "require('./node_modules/@phlix/ui/package.json').version"`. If stale, force re-resolve with `npm install "@phlix/ui@github:detain/phlix-ui#vX.Y.Z" --save`, then re-check the version AND that new seam exports exist (`grep -oE 'seekBy|playLocalFile' node_modules/@phlix/ui/dist/index.d.ts`).
- **[gotcha]** GitHub Actions artifacts are ALWAYS served zipped — there is no setting to offer a bare clickable file. To provide a raw downloadable `.exe`/`.appx`, you must attach it to a GitHub **Release** (e.g. a rolling "latest" prerelease), not an upload-artifact step.
- **[pattern]** Rolling "latest" release in `build.yml`: a `release-latest` job (gated `if: github.event_name == 'push' && github.ref == 'refs/heads/master'`) that `actions/download-artifact`s the built installers, runs `gh release delete latest --yes --cleanup-tag || true` (so the tag re-points to the new commit and no stale assets accumulate), then `softprops/action-gh-release@v3` with `tag_name: latest`, `prerelease: true`, `make_latest: false`, `target_commitish: ${{ github.sha }}`. The job needs `permissions: contents: write`.
- **[convention]** `build.yml` has three-way gating: the `build` job runs on push/PR/tag; the `release-latest` job runs ONLY on master pushes (skips PRs and tags); the `release` (versioned) job runs ONLY on `refs/tags/v*`. Preserve these guards when adding triggers so PR/master builds publish downloadable assets without cutting a versioned release.
- **[gotcha]** `createPhlixApp` renders NO top-bar nav at all unless you pass a `menu: MenuItem[]` — the admin section (and Browse/Settings) is unreachable in the Windows client without it. The gated "Admin" entry needs `{ id:'admin', label:'Admin', to:'/app/admin/dashboard', requiresAdmin: true }` (the shell shows it only for `useAuthStore().isAdmin`); pair it with `extraRoutes: [...buildAdminRoutes()]` (server) or `...buildHubAdminRoutes()` (hub) so the route actually resolves. Build these per app-mode (`buildMenu`/`buildExtraRoutes`) since hub vs server have different surfaces.
- **[pattern]** Before wiring any `@phlix/ui` page/route builder into the renderer, confirm it's actually exported in the INSTALLED build: `grep -oE 'buildAdminRoutes|buildHubAdminRoutes|LibraryScanPage|MyServersPage|FederationPage|ManageSharesPage|MenuItem' node_modules/@phlix/ui/dist/index.d.ts | sort -u`. The pinned tag may lag the upstream source.
- **[gotcha]** In hub mode, do NOT let the app land on the media-server Browse page — its server-only endpoints 404 on the hub. Pass `home: '/app/servers'` and `features: { resumeSync: false }` (skip continue-watching) for `app: 'hub'`; server mode keeps the default Browse home.
- **[gotcha]** Route paths passed to `createPhlixApp`'s `extraRoutes` carry the FULL `/app` prefix (e.g. `/app/admin/dashboard`, `/app/servers`) because the router history base is `/`, not `/app`. Menu `to:` targets must match.
- **[pattern]** When `main.ts` grows its `@phlix/ui` import surface, the `vi.mock('@phlix/ui', …)` in `tests/unit/main.test.ts` must stub every newly-imported builder/component (return marker route arrays for builders, `{ template:'<div/>' }` page stubs) or the test import throws. Also switch the `createPhlixApp` call assertion from `toHaveBeenCalledWith({…})` to `expect.objectContaining({…})` so adding new config keys (menu/extraRoutes/home/features) doesn't break the exact-match.
- **[gotcha]** Windows CI `EBADENGINE` warnings (`electron@42` / `@electron/get` / `@electron/rebuild` / `@babel 8` want node `>=22.12`) are WARNINGS, not failures — `npm install` still succeeds. The fix is to bump the runner Node, not chase the install exit code.
- **[convention]** The renderer build (`tsconfig.json`) is NOT clean on master — a green `npm test` (vitest) + `npx eslint .` (exits 0) is the real pre-commit signal, not a zero-error renderer tsc. Full local PR gate: `npx vue-tsc --noEmit && npx tsc -p tsconfig.main.json --noEmit && npx vitest run && npx eslint .` (all four EXIT=0 on a clean change). Note `tsc -p tsconfig.main.json --noEmit` (main/preload) IS clean and prints nothing on success.
